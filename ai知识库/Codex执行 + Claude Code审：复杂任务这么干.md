# Codex执行 + Claude Code审：复杂任务这么干

> 作者：FredTalk
> 时长：约3分20秒
> 链接：[https://v.douyin.com/uNSq6SkrHuQ/](https://v.douyin.com/uNSq6SkrHuQ/)

***

## 核心观点

Codex 搭配 Claude Code 能让 AI 输出质量提升 50% 以上。核心原因不是哪个工具更聪明，而是**不要让同一个 AI 自己做完再自己审**。把"做事的人"和"挑错的人"分开，很多原来漏掉的问题反而更容易冒出来——就像你自己写一篇文章马上自己改，很多错别字和逻辑问题也看不出来。

***

## 为什么"自己审自己"不行？

### AI 的盲区

| **现象**             | **原因**             |
| ------------------ | ------------------ |
| 顺着思路一直做下去          | 任务一复杂，AI 容易陷入自己的逻辑 |
| 检查时继续相信这条路         | 刚做完时会觉得这条路是对的      |
| 不是不认真，是自己看自己本来就有盲区 | 和人类一样，AI 也有"确认偏误"  |

> 就像你自己写一篇文章马上自己改，很多错别字和逻辑问题也看不出来。AI 也一样——他刚刚会觉得这条路是对的，检查的时候就很容易继续相信这条路。

***

## 角色分工

### Codex：执行者（做事的人）

| **职责** | **具体工作** |
| ------ | -------- |
| 读项目规则  | 理解约束条件   |
| 改文件    | 实际编码/修改  |
| 跑检查    | 执行验证流程   |
| 做出第一版  | 输出初版成果   |

### Claude Code：审核者（挑错的人）

| **职责** | **具体工作** |
| ------ | -------- |
| 站在外面看  | 不着急继续写   |
| 专门问问题  | 质疑、挑毛病   |
| 不干活只挑错 | 角色必须明确   |

> 如果两个 AI 都在赶活，你很快就不知道到底谁改了什么。先把问题找出来，然后再决定到底应该怎么改。

***

## 完整四步工作流

### Step 1：目标说清楚，Codex 做出第一版

把目标描述清楚，让 Codex 执行并输出初版成果。

### Step 2：Codex 写过程文档

让 Codex 把以下信息写下来：

* 改了什么
* 怎么验证的
* 哪里不确定

> **关键技巧**：不是多开几个 AI 窗口就能解决，而是有没有把关键的过程文件留下来。如果材料没有留存，另一个 AI 其实也只能够重新拆一遍。

**过程文档示例**：

* `Plan.md`：计划文档
* `Progress.md`：进度文档
* `QC.md`：审查报告
* 交接文档：同时告诉 AI 要做什么、改了什么、怎么检查

### Step 3：Claude Code 专门审核

把 Codex 的输出和过程文档交给 Claude Code，让它站在外部视角审核。

**审核四问**：

| **#** | **问题**    | **能揪出的问题** |
| ----- | --------- | ---------- |
| 1     | 有没有跑题？    | 偏离原始需求     |
| 2     | 规则有没有漏看？  | 遗漏约束条件     |
| 3     | 是不是真的检查过？ | 假检查、走过场    |
| 4     | 下一步清不清楚？  | 后续计划模糊     |

> 这四件事情看起来很简单，但是很多看起来已经完成的毛病，基本上都能从这里揪出问题来。

### Step 4：问题清单 → 代办事项 → 逐条处理

| **动作**      | **说明**                 |
| ----------- | ---------------------- |
| 审核完后        | 不要让 Claude Code 自由发挥修改 |
| 把问题清单变成代办事项 | 结构化问题列表                |
| 一条一条处理      | Codex 按清单逐条修改          |
| 每改完一条跑对应验收  | 验证修改效果                 |
| 把结果写回文档     | 更新过程文档                 |
| 直到最终需求做完    | 闭环管理                   |

> 这样就不是无限地来回扯，而是一轮一轮地把问题解决掉。

***

## 关键技巧总结

| **技巧**        | **说明**                            |
| ------------- | --------------------------------- |
| **过程文档必须留存**  | 任务短时靠聊天记录还能勉强记住，任务一复杂上下文会乱、记忆也会压缩 |
| **角色必须说清楚**   | Claude Code 只挑问题不要干活              |
| **审核四问是标准动作** | 有没有跑题、规则有没有漏看、是不是真的检查过、下一步清不清楚    |
| **问题清单化**     | 审核完后把问题变成结构化代办，逐条处理               |
| **改完即验**      | 每改完一条就跑对应验收，不要攒到最后一起验             |

***

## 实战演示：生成数据看板 HTML

**流程**：

1. 让 Codex 生成数据看板 HTML
2. 发给 Claude Code 审核
3. Claude Code 给出结论（如：三个明显 bug）
4. 把 Claude Code 的建议写进 `QC.md`
5. 让 Codex 按照 `QC.md` 里的建议逐条修改
6. 输出质量和结果大幅提升

***

## 思考与延伸

1. **"执行-审核"分离是软件工程的基本原则**：代码审查（Code Review）是软件行业的最佳实践，FredTalk 把这个原则应用到了 AI 工作流中——AI 也需要 Code Review。
2. **过程文档是 AI 协作的"契约"**：没有文档，两个 AI 之间的协作就是黑盒；有了文档，协作就有了可追溯、可验证的基础。
3. **审核四问的普适性**：这四问不仅适用于代码审核，也适用于任何复杂任务的验收——跑题、漏规则、假检查、下一步不清，是绝大多数"看起来完成了"的任务的通病。
4. **"角色混淆"是 AI 协作的最大陷阱**：如果两个 AI 都在"干活"，很快就不知道谁改了什么。明确分工是高效协作的前提。
5. **从"聊天"到"文档驱动"的范式转变**：短任务靠聊天记录还能应付，复杂任务必须文档驱动——这是 AI 工作流从"玩具"走向"生产工具"的必经之路。

## 完整文稿

> 我敢说 Codex 搭配 Claude Code 能够让你的 AI 输出质量提升 50% 以上。今天 Fred 就给大家讲一个我一直都很常用的工作流：Codex 先负责把事情做出来，而 Claude Code 负责去挑毛病。不是因为其中一个会比另外一个聪明多少，而是当你的任务一复杂，一个 AI 很容易顺着自己的思路一直做下去。换一个工具专门来进行审核，很多原来漏掉的问题反而更容易冒出来。

> 就像你自己去写一篇文章马上自己改，很多错别字和逻辑问题你也看不出来。AI 也一样——他刚刚会觉得这条路是对的，检查的时候就会很容易继续相信这条路。所以说不是他不认真，而是自己看自己本来就会有盲区。

> 所以我的做法很简单：把做事的人和挑错的人分开。Codex 更适合先执行——比如说读项目的规则、改文件、跑检查、做出第一版。而 Claude Code 就站在外面看，不要着急继续写，而是专门去问问题。这两个角色一分开，结果就会好很多。

> 完整流程一般是这个样子：
>
> 1. 先把你的目标说清楚，让 Codex 做出第一版。
> 2. 让 Codex 把他改了什么、怎么验证、以及哪里不确定都写下来。
> 3. 让 Claude Code 专门去审核这一版。审完以后不是让他重新去修改，而是把问题列出来。
> 4. 最后 Codex 照着问题一条一条去修改。

> 这样就不是无限地来回扯，而是一轮一轮地把问题解决掉。

> 这里有一个很关键的技巧：不是多开几个 AI 窗口就能解决，而是有没有把这种关键的过程文件留下来。如果材料没有留存，另外一个 AI 其实也只能够重新拆一遍。我一般会把这些东西写进文档里面，比如 Plan.md、Progress.md、QC 的审查报告和交接的文档，同时告诉 AI 要做什么、改了什么以及怎么检查。任务短的时候靠聊天记录还能够勉强记住，任务一复杂上下文会乱、记忆也会压缩。

> 第二个很重要的点就是要把 Claude Code 的角色说清楚——只挑问题不要干活。因为如果两个 AI 都在赶活，你很快就不知道到底谁改了什么。先把问题找出来，然后再决定到底应该怎么改。

> 具体审核的时候，我会让他问四件事情：
>
> 1. 有没有跑题？
> 2. 规则有没有漏看？
> 3. 是不是真的检查过？
> 4. 下一步清不清楚？

> 这四件事情看起来很简单，但是很多看起来已经完成的毛病，基本上都能从这里揪出问题来。

> 审核完了以后也不要让 Claude Code 自由发挥，把问题清单变成代办，然后一条一条处理。每改完一条就跑对应的验收，再把结果写回文档，直到最终需求做完。

> 我给大家演示一个：比如我让 Codex 去帮我生成一个数据看板 HTML，生成完了之后我就发给 Claude Code 让他帮我去进行一个对应的审核。然后他就会一个一个给到一些结论，比如说给到说有三个明显的 bug。这个时候我就只需要发给 Codex，让他帮我按照 QC.md 里面 Claude Code 提的建议进行修改和调整。这样输出的质量和结果就会好很多。
