# 一个Skill文件里应该有哪些东西

> 作者：宇哥AI职场落地-培腾AI
> 时长：约2分5秒
> 链接：[https://v.douyin.com/U\_0VH\_NLPe8/](https://v.douyin.com/U_0VH_NLPe8/)

***

## 核心观点

很多人做 Skill 时里面只有一个 `skill.md` 就完事了，但作者强调这离一个专业级的 Skill 差得太远。一个真正的专业 Skill 需要分层级、成体系的文件结构——**入门版靠嘴说，标准版给文档，增强版给脚本，达到产品级就要有一个体系。**

***

## Skill 文件的四层架构

### 第一层：核心大脑（必备）

| **文件**       | **作用**                                     |
| ------------ | ------------------------------------------ |
| **skill.md** | Skill 的核心文档，相当于 Skill 的大脑。没有它，AI 根本不知道该怎么干 |

### 第二层：标准版（案例 + 规则）

| **文件**      | **作用**                                                |
| ----------- | ----------------------------------------------------- |
| **example** | 案例——你的输出案例大概是什么样子                                     |
| **rules**   | 规则和约束——给 AI / Skill 一个规则，知道什么时候开始调用、什么时候不能调用、什么情况可以调用 |

### 第三层：增强版（脚本 + 模板 + 资源）

| **文件**        | **作用**                   |
| ------------- | ------------------------ |
| **scripts**   | 执行脚本                     |
| **templates** | 可复用的模板（如输出模板）            |
| **assets**    | 资源文件——类似于一个小知识库，比如需要查的资料 |

> 到了这一层，你的 Skill 基本上就有了"共同能力"（可复用、可扩展）。

### 第四层：产品级（测试 + 配置 + 版本管理）

| **文件**     | **作用**                |
| ---------- | --------------------- |
| **test**   | 测试文档——你做过的测试记录        |
| **config** | 配置文件——比如口吻怎么追的、参数设置   |
| **readme** | 版本更迭文档——每一次优化都可以记录在这里 |

### 进阶版：环境配置 + 多 API 备份

| **文件**       | **作用**                       |
| ------------ | ---------------------------- |
| **.env**     | 环境变量文件——比如调用 Skill 时的 API 密钥 |
| **多 API 配置** | 不能只用单一 API，出错时要有备份方案，可配置、可移动 |

***

## 总结：四个层级对应四个段位

| **段位**  | **特征** | **核心交付物**                                  |
| ------- | ------ | ------------------------------------------ |
| **入门版** | 靠嘴说    | 一个 skill.md                                |
| **标准版** | 给文档    | skill.md + example + rules                 |
| **增强版** | 给脚本    | + scripts + templates + assets             |
| **产品级** | 成体系    | + test + config + readme + .env + 多 API 配置 |

***

## 为什么分层很重要？

1. **减少算力消耗**：结构化的 Skill 让 AI 更快理解任务，减少不必要的推理
2. **降低出错率**：明确的规则和约束减少了 AI "自由发挥"的空间
3. **方便修改维护**：分层后，改规则不用动核心逻辑，改模板不用改脚本
4. **可复用可扩展**：templates 和 assets 可以被多个 Skill 共享
5. **版本可追溯**：readme 记录每次优化，便于回溯和协作

***

## 思考与延伸

1. **你的 Skill 在第几层？** 对照这个分层模型，检查你正在使用的 Skill 是否只有"入门版"的水平。
2. **为什么 example 比 prompt 更重要？** 给 AI 一个具体的输出案例，比写100行描述性 prompt 更有效——"show, don't tell"。
3. **rules 的边界意识**：一个专业的 Skill 必须明确"什么时候不调用"——边界清晰的 Skill 比万能的 Skill 更可靠。
4. **产品级 Skill 的工程化思维**：test + config + readme 是软件工程的基本要素，AI Skill 正在从" prompt 艺术"走向"工程化产品"。
5. **多 API 配置的鲁棒性**：依赖单一 API 的 Skill 在生产环境中是脆弱的——产品级 Skill 必须有 failover 机制。

## 完整文稿

> 很多兄弟们在做 Skill 的时候，里面就一个 skill.md 就完事了。那我想告诉大家，这个离一个专业的 Skill 差得太远太远。
>
> 那么一个真正的专业级的 Skill 里面该有哪些东西？以后你在让 AI 去帮你写 Skill 的时候，你就可以把这个格式发给它。这样不仅能减少算力，还能减少 AI 的出错率，还更好修改、方便后期维护。
>
> 大家先看这张图，看完这张图之后我来给大家详细讲。
>
> **第一层**，就是我们所有人都知道的，放一个 skill.md 这个文档。这个文档就相当于你这个 Skill 的大脑，如果说没有它，那么它根本就不知道该怎么去干。
>
> **第二层**，标准版有两个东西，一个叫 example，第二个叫 rules。主要有这两个东西。第一个不用说了，是案例——你的输出案例大概是什么样子。第二个是规则和约束，也就是说给这个 AI、给这个 Skill 一个规则，知道什么时候开始调用、什么时候不能调用、什么情况可以调用。
>
> **第三层**，是额外的三个。第一个 scripts 执行脚本，第二个 templates 放你可以复用的模板，就比如说输出模板。第三个放 assets，什么意思？这个是资源文件，就比如说我需要去查出什么样的资料，就是类似于一个小的知识库。到了这一层，你的 Skill 基本上就有共同能力了。
>
> **第四层**，也有三个。第一个是 test，就是你做了一些测试文档。第二个叫 config，主要放的是一些配置文件，比如说这口吻怎么追的。第三个就是很重要的一个 readme，就是你可以做版本更迭，然后你的每一次优化都可以放到这个文档里面。
>
> 但是也不够。甚至刚刚说了还有一些东西，比如说 .env 文件。那么就是比如我调用这个技能的时候需要 API，那么这个 API 我不能只用一个，那如果说出错了它肯定有一个兜底，那我可以放两个 API。那这个就是可配置的、可移动的。
>
> 当然还有一些进阶版更详细的资料，我给大家列好了所有的包含的东西，包括它每一层是什么意思，里面还有一些关于 Skill 的介绍、补充介绍，和一个最基础的 Skill 包含的东西。为什么包含这些东西？用一种很通俗的语言给大家整理好了。这个文档很适合入门新手，需要的可以说一声，给大家分享。
>
> 总之总结成一句话：**入门版靠嘴说，标准版给文档，增强版给脚本，达到产品就要有一个体系了。**
