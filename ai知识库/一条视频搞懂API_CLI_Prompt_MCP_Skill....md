## 一、核心观点与总结

### 核心论点

AI领域的6个核心概念——API、CLI、Prompt、MCP、Skill、Agent——之间存在一条清晰的演化路线：**每一步都在降低门槛，把原本只有专业人士才能用的能力逐渐交到更多普通人手里**。理解了这条演化链，以后无论出现什么新的AI词汇，都能对号入座。

### 6个概念详解

#### 1. API（Application Programming Interface）

* **定义**：程序和程序之间沟通的语言
* **类比**：餐厅的菜单——你通过菜单点菜，厨房根据菜单做菜，你不会直接跑进厨房告诉厨师
* **本质**：一个程序想调用另一个程序的能力，通过API协议实现
* **出现时间**：大模型之前就已存在

#### 2. CLI（Command Line Interface）

* **定义**：人和程序之间沟通的语言
* **与API的区别**：API是程序↔程序，CLI是人↔程序
* **特点**：在黑框框（命令行终端）里通过命令操作软件，面向专业和技术人员
* **共同点**：API和CLI的"I"都是Interface（接口）

#### 3. Prompt（提示词）

* **定义**：给模型规范和约束，让它按你的需求来输出
* **作用**：限制模型的输出格式（如文章结构、语气、风格等）
* **局限**：光靠Prompt，模型无法查路线、查天气、下订单等需要外部能力的任务

#### 4. MCP（Model Context Protocol）

* **定义**：让不同模型厂商和工具服务商之间有一个统一的协议来沟通
* **类比**：就像USB统一协议出现之前，每家手机用自己的数据线；MCP就是"USB统一协议"
* **结构**：MCP Client（工具的调用方）+ MCP Server（工具的提供方）
* **解决的问题**：各家各自订协议、重复造轮子、开发成本极高
* **两大缺陷**：
  1. 工具信息全部加载到模型上下文，**挤占上下文空间**，导致幻觉概率增加、工具调用不准
  2. 通过MCP实现的工作流程**无法复用**，A实现一套、B实现一套

#### 5. Skill（技能）

* **定义**：把标准SOP程序化的操作手册
* **类比**：汽车修理店的维修手册 / 公司的入职流程文档
* **三大优势**：
  1. **自然语言描述SOP**：用自然语言表达需求
  2. **按需加载**：只加载Skill的描述信息到上下文，不加载全部内容，解决MCP的上下文空间问题
  3. **可复用分享**：可以把Skill分享给别人，大家不用重复开发
* **重要意义**：Skill是AI发展的一个基点，真正让AI走到普通人面前，实现AI平权

#### 6. Agent（智能体/代理）

* **定义**：把事情交给它，它来帮你完成，你只需验收结果
* **三大组成部分**：

| **组件**         | **作用**                   |
| -------------- | ------------------------ |
| **LLM（大模型）**   | Agent的核心大脑               |
| **Memory（记忆）** | 记住过往历史、对话记录、知识库          |
| **Tools（工具）**  | API、CLI、MCP都是Agent可调用的工具 |

* **Agentic Loop（代理循环）**：收集上下文 → 执行 → 校验结果 → 不满意则继续循环（收集→执行→校验），直到给出满意结果

### CLI vs MCP：谁取代谁？

**不是取代，而是共存。** CLI调用简单、非常适合模型调用，很多软件本身就是CLI化的，不需要再封装MCP Server。CLI是一种更方便、更便捷、更低成本的接入Agent的方式。未来MCP和CLI会共存，配合Agent实现更多能力。

### 演化路线图

```
API（程序↔程序）→ CLI（人↔程序）→ Prompt（人↔模型）→ MCP（模型↔工具，统一协议）→ Skill（可复用的操作手册）→ Agent（自主完成任务的代理）
```

**每一步都在降低门槛**，从专业人士到普通人的AI平权之路。

## 三、视频逐字稿

今天一条视频帮你搞懂什么是API、CLI、Prompt、MCP、Skill、Agent。太阳底下无新先事，教你搞明白这6个概念，不管以后出现什么样的AI词汇，你都能对号入座了。

首先我们介绍两个大模型出现之前就已经存在的概念。

**第一个API**。什么是API呢？API就是程序和程序之间沟通的语言。一个程序想调用另外一个程序的能力，它们之间就是通过API来实现的。给大家打个比方，比如你要去一家餐厅吃饭，首先你是不是要通过菜单来进行点菜，然后厨房是根据你的菜单来决定给你做什么样的菜。而这个菜单就是你和餐厅之间沟通的语言，你不会直接跑到厨房里面去告诉厨师你想吃什么，那样的话整个餐厅可能就乱套了。这个菜单就是你和餐厅之间沟通的语言，对应的API就是程序和程序之间沟通的语言。比如我要调用大模型的API，都是通过API这个协议来实现的。

**接下来CLI**。API和CLI后面都有一个"I"，这个I就是Interface接口。API是程序和程序之间的沟通语言，那CLI就是人和程序之间的沟通语言。大家操作软件的时候通常都是有界面的，你点一个按钮打开一个页面，但是对于专业技术人员来讲，通过界面的操作不一定是最方便最高效的方式，反而CLI这种在黑框框里面命令行的终端，在里面通过命令来操作软件反而是更高效的。所以它其实是面向专业和技术人员来使用的，人通过命令来操作程序或软件，这就是CLI的作用。

**大模型到来了，我们首先接触的概念就是Prompt提示词**。我们最开始用大模型的时候通常拿它做什么事情呢？比如写稿、翻译、写代码。比如我们要写一篇稿子，我们会通过提示词来限制它：开头要怎么写、文章结构是什么样的、结尾是怎么样的、按照什么样的语气来进行表达。这些其实都是通过提示词给模型一些规范和约束，让它按照你的需求来进行输出。这就是提示词本质的作用。

但是我们提示词用着用着就会发现还缺了一点东西。比如我想通过模型去查路线、查天气、下订单该怎么办呢？早期模型光靠自身的能力没有办法实现这个事情，所以接下来我们需要让模型可以跟外部的工具去连接，让它可以去调用工具来解决我们现实生活工作中的问题，极大地拓展模型的能力。这时候工具就出现了。

工具出现之后，我们发现有非常多的大模型厂商，服务工具的提供商也非常多，大家早期都是各自订各自的协议，开发成本极高，大家都在重复造轮子。就像我们早期USB协议没有出来之前，大家都是用自己的协议来适配手机、适配各种设备，最终你可能发现身上需要带一堆的数据线。这就是没有统一协议的坏处。

**MCP的诞生就是来解决这个问题**，让不同的模型厂商和工具的服务商之间可以有一个统一的协议来互相沟通和交流。MCP的P就是Protocol，代表协议。既然是协议，其实跟生活中碰到的合同协议是一个意思。比如我要跟一家公司合作，我付他钱，他给我提供服务，我们之间就需要签订一个协议，大家按照协议中写的内容去执行。对于MCP也是一样，它也有双方：一方是MCP的Client（工具的调用方），一方是MCP的Server（工具的提供方），他们通过MCP协议来进行交互。这样我们就解决了模型和外部工具连接的问题。

但是用着用着，它会出现一个非常大的缺陷，涉及到上下文的概念。大家都知道模型上下文的空间是有限的，从200k到100万token。MCP的Server存在的问题是：通常要实现一个Agent，它需要对接的工具非常多，一个MCP的Server少则几十个，多则集成了上百个工具。这些工具在使用的时候，所有信息都会被加载到模型的上下文空间里面去，极大地占据上下文空间，导致真正留给业务逻辑的空间就少了。这会带来两个问题：第一，模型出现幻觉的概率增加了；第二，工具调用就不准了，你本来想调用A工具，实际上给你调用到B工具上去了。

但这还不是最根本的，最根本的是说我们通过MCP去实现的一些工作流程是没有办法被复用的。A实现了一套，B实现一套，C实现了一套，他们都在重复造轮子。

**所以后面Skill出现了**。Skill简单理解就是一个操作手册。比如你是一家汽车修理店的修理工，手册上有非常多的工具可以使用，你现在需要去修一辆车，这个时候你需要一个操作手册，按照这个操作手册一步一步来做。再换一个例子，比如你加入一家公司，公司会给你提供入职流程，团队也有项目的流程，你按照这个流程就可以顺利入职上手工作。Skill就是这样一个操作手册，它其实是把标准的SOP给程序化了。

它带来的几个好处：第一，你可以通过自然语言描述你的SOP来表达你的需求；第二，它是按需加载的，不管你安装了多少个Skill，它只是把Skill里面的描述信息加载到上下文，而不是把全部内容加载到上下文，解决了MCP挤占上下文空间的问题；第三，它是可以复用的，你可以把自己的Skill分享给别人，让大家都可以用。这也是现在Skill如此流行的一个很重要的原因，大家都可以去一些GitHub或其他地方找到适合自己的Skill，别人已经开发好了，你不需要自己再开发。

我觉得Skill的出现其实是AI发展的一个基点。无论我们刚才讲到的API、CLI还是MCP，它们其实都是接口层，从这个词上来讲都是普通人很难理解的，过去还是变相的专业人士和程序开发人员的专利。Skill呢，其实是真正让AI走到了普通人面前，让普通人也可以使用AI的能力来解决自己工作和生活中的问题，真正带来AI的平权。

**接下来介绍Agent的概念**。Agent从字面意思来讲就是代理。代理是什么？代理是说把事情交给它，它来帮你完成，你不需要自己去关心过程中的实现细节，你只要学会怎么验收结果就行了。

一个Agent由三部分组成：

* **LLM（大模型）**：Agent的核心大脑
* **Memory（记忆）**：记住过往历史、对话记录、知识库
* **Tools（工具）**：API、CLI、MCP都是Agent可以调用的工具

关于CLI是否会取代MCP的问题：并不是说CLI会取代MCP，而是说CLI是一种更方便、更便捷、更低成本的接入Agent的方式。未来MCP、CLI都会共存，一起配合Agent去实现更多的能力。

**Agent的工作流程——Agentic Loop**：简单来说就三步——第一步收集（收集上下文，包括你的输入、从网上获取的信息、知识库），第二步执行，第三步校验结果。如果觉得结果不满意，还会继续循环：收集更多信息→再执行→再校验，不断循环，最终给你一个比较满意的结果。

这6个概念讲完了，你有没有发现它们之间有一条非常清晰的演化路线：从程序和程序之间的沟通语言API，再到人和程序之间的CLI，再到大模型时代的Prompt、MCP、Skill和Agent，每一步其实都是在降低门槛，把原本只有专业人士才能用的能力逐渐交到更多普通人的手里。
