## 一、核心观点与总结

### 核心论点

Browser Harness 是 Browser Use 团队开源的一个仅592行Python代码的项目，它通过 WebSocket 直连 Chrome DevTools Protocol (CDP)，彻底拆除了 Agent 与浏览器之间的所有抽象层，让 AI Agent 第一次真正拥有了浏览器的「完全控制权」。8天获得6700星，其背后是一场关于 AI Agent 自由度与安全性的终极博弈。

### 传统浏览器自动化的三大痛点

1. **脆弱性**：页面按钮文字变化，CSS选择器就失效；Click返回成功但什么都没发生
2. **昂贵性**：每次操作都要把完整页面快照喂给大模型，复杂页面上万Token
3. **局限性**：传统框架本质是给Agent发操作手册（Click/Type/Scroll），手册没写的Agent做不了

### Browser Harness 的革命性设计

### 最震撼的能力：自律循环（Self-Healing）

当Agent发现缺少某个工具时，它会：

1. 打开 `helper_store.py`
2. 发现没有对应函数
3. **自己写一个函数**
4. 重新运行

典型案例：开发者忘了实现文件上传功能，Agent遇到文件输入框后，自己查阅CDP文档，用 `DOM.setFileInputFiles` 写了一个上传函数——开发者看GitHub PR才知道多了7行代码。

### 三大安全风险

1. **财务操作风险**：Agent操作的是真实浏览器，没有任何内置白名单
2. **提示注入风险**：网页恶意内容可能被Agent执行，且Agent会主动修改自己的代码
3. **静默作恶风险**：Agent执行了你不知道的操作且成功了，Git只记录代码不记录浏览器操作

但让硅谷真正炸锅的，是他把Agent和浏览器之间的每一层抽象全部拆掉了。

要理解为什么重要，先看看之前Agent操作浏览器有多难受。如果你用过Selenium、Puppeteer、Playwright，你一定见过这些错误：页面按钮文字变了，选择器就废了；更恶心的是静默失败——Click返回成功，但什么都没发生。还有Token烧钱，每次操作都要把完整页面快照喂给大模型，一个复杂页面上万Token。

传统浏览器自动化：脆弱、昂贵、局限。

但更深的问题是，围墙太多。传统Agent的框架，本质上都是给Agent发一本操作手册。手册上写了Click、Type、Scroll，手册上没写的Agent做不了。这就是Agent面前的第一道墙——预设的工具边界。

Browser Harness做的事情，就是把这道墙拆了。

Browser Harness创始人Gregor Zunic说：每一个抽象都是负债，每一个辅助模块都是故障点。框架失败不是因为模型弱，而是因为行动空间不完整。
