## 三、视频逐字稿

Anthropic 又又又刷屏了。就在你开心自己花了三个月，终于把一套 AI 工作流的框架搭出来的时候，Claude Code 直接发布了一套面向所有任务的动态工作流，标题就一句话：给每个任务配一个 Harness。你可能摆干了。

先讲一个让人后背发凉的数据。SWE-bench 原班人马搞了个新测试叫 ProgramBench，只给 AI 一个编译好的程序加一份文档，让它从零把整个代码库重写出来，复现原程序的行为，不准看原码不准连网。结果，九个一线模型，Opus、GPT 全在内，能完整重建的一个都没有，通过率零。

这不代表模型不会写代码。在真实工程里，顶级团队给模型配上环境、工具、反馈和验证回路，三个人能驱动它写出百万行级别的代码。中间这个差距就是 Harness。

Harness 质疑就是码距。模型是一匹宝马，你控制不住它，跑得再快也没意义。业界有句话特别精辟：Agent 等于 Model 加 Harness。模型决定它能多强，Harness 决定你能不能驾驭它。

放到眼睛脉络里看，最内层是提示子工程，关心怎么给 AI 下指令；中间是上下文工程，关心给它什么信息；最外层的 Harness 工程，把这两层全包进去，再加上工具编排、状态持久化、验证循环、任务拆解、子 Agent、全线沙箱回滚机制，形成一整套工程基础设施。

但很多传统 Agent 的框架，尤其是那种主 Agent 一边规划、一边调度、一边读回所有结果的模式，都会遇到一个老问题——上下文爆炸。因为发号施令的始终是 Claude 本身，每个小弟看完结果都得先塞回它的脑子里，它读完才能决定下一步。任务一大，这个脑子自己先爆了，它开始偷懒、开始偏袒自己、开始忘掉你一开始的要求。

Dynamic Workflow 的破局点就在这。它把编排这件事，从 Claude 的那个会爆的脑子里，整个搬进了一段独立的代码。Derek 这次有句话说得很狠：框架的形状，应该顺应任务的形状。人不可能预先写死一个完美通用的脚本，最理想的状态是来什么任务，现场给它组装一套对应的 Dynamic Workflow。

Dynamic Workflow 就是 AI 针对当前这个任务，实时生成的一套调度逻辑。它强在三个地方：

第一，它把计划变成了确定性的代码。面对大任务，Claude 的第一步不是埋头干，是先写一段 JavaScript 脚本，哪里分 20 个子 Agent，哪里调哪个验证器，全写死在代码里，外部运行时严格按脚本执行，用确定性的程序框住概率性的模型输出。

第二，它自带对抗验证。Dynamic Workflow 会自己组织起来，A Agent 写代码，B Agent 专门当敌人去挑刺，甚至调更强的 Opus 来审，过不了脚本里定义的验证闭环，代码就别想往下走。

第三，它能检查点容灾。传统聊天挂了就得重来，Dynamic Workflow 有检查点，哪怕跑了三小时、派了几百个子 Agent 中途被叫停，它记着每一步的执行状态，下次接着跑。

用法简单到离谱。你直接在 prompt 里写关键字 workflow 就行，或者用 Claude Code 的命令。你想想看这几个 prompt：翻我最近五十次绘画，把我反复犯的错挖出来，写进规则文件。你甚至能给它设预算，直接说这个任务只准用一万 token。

Claude 写 Workflow 的时候，会反复用上几种固定套路。记住这几个，你就能看懂它在干嘛：

最常见的是拆出再综合——把一个大任务拆成一堆小任务，每步派一个 Agent，最后汇总。还有锦标赛——不分工，让一堆 Agent 用不同方法做同一件事，两两 PK 选出冠军。再就是对坑验证——每个产出都派一个敌人去挑刺。还有生成再过滤——先狂想一堆点子，再按标准筛掉。这几种还能像积木一样拼起来用。

把这套东西用到极致的是 Bun，一个 JS 开发工具。六天前，Bun 的创始人 Jarred 还在 Hacker News 上说，这堆 Rust 的代码根本跑不起来，最后大概率全部扔掉。六天后，几十万行代码从 Zig 整个重写成 Rust，测试通过率 99.8%。Jarred 自己说，就是 Dynamic Workflow 加对抗式代码审查这两样，才让六天重写 Bun 变成可能。不是某人用 AI 写了几个函数，是 Claude 自己写的一套规章制度，组织了一整支队伍。

这套东西最猛的用法，其实不再写代码。80 份简历，它先排序，再复合前十，一份提案，同时拉来投资人、客户、对手三个视角互相拷打。Slack 里没人整理过的历史记录，它能挖出来。命名这种事，先狂想一堆，再两两 PK 选前三。事实声明，逐条核查。大规模工单自动分流。凡是需要大量并行加严格验证的活都能上。
