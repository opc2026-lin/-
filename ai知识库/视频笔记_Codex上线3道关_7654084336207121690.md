### Codex 上线前的三道质量关

#### 第一关：逻辑一致性审查

**AI 代码最常见的问题不是语法错误，而是逻辑不自洽。**

- Codex 生成代码时是"接龙式"思维，它只看上下文概率最高的下一段代码，不关心全局逻辑

#### 第二关：边界与异常测试

**AI 擅长写"正常路径"，但不擅长处理"异常情况"。**

- Codex 生成的代码往往只覆盖"Happy Path"（理想情况），缺少空值检查、类型判断、异常捕获

#### 第三关：安全与依赖审查

**AI 会引入你根本没注意到的风险和债务。**

- Codex 可能在代码中引入过时或存在漏洞的第三方依赖
- 可能生成硬编码的 API Key、密钥或敏感配置
- 可能引入不必要的权限请求（如过度请求文件系统或网络权限）
- 必须做的事：**逐行检查敏感信息**，核对 `package.json`/`requirements.txt` 中的依赖版本是否安全，确认没有被注入不必要的权限

### 三道关的实操 Checklist

| 关卡       | 检查项                  | 不检查的后果     |
| -------- | -------------------- | ---------- |
| **逻辑审查** | 全局调用链是否闭合、业务逻辑是否前后一致 | 功能残缺、逻辑混乱  |
| **边界测试** | 空值/异常/超长输入处理、并发安全性   | 线上崩溃、数据错乱  |
| **安全审查** | 硬编码密钥、依赖漏洞、过度权限      | 安全漏洞、供应链攻击 |

Codex 不是不能用，而是不能**不审就用**。把它当成一个超级实习生：代码写得快，但经验不足，需要你（资深开发者/产品Owner）逐道把关。改完代码 → 过三道关 → 测试通过 → 再上线。顺序不能乱，一步都不能省。
