# VibeCoding最佳实践：极简工作流

> 作者：Edward\_ai
> 时长：约1分43秒
> 链接：[https://v.douyin.com/1Q0GjkxTWVs/](https://v.douyin.com/1Q0GjkxTWVs/)

***

## 核心观点

为什么你的 VibeCoding 出来的是玩具，大佬做出来的却是成熟产品？**很大的原因在于工作流程有问题。** 作者在经历两场黑客松、一场创意比赛后，总结出了一套极具普适性的极简工作流：想法 → 压力测试 → PRD → 需求对齐 → 设计基调 → 开发 → 版本管理 → 测试 → 重构。

***

## 极简工作流五步法

### Step 1：压力测试想法

**Skill：Star Pressure Test**（或 Claude 的类似 Skill）

> 不要一有想法就马上开始动工。很多想法在刚开始时都是有问题的。

用 Star Pressure Test 对想法进行压力测试，找出想法中的漏洞和风险点。

### Step 2：生成产品需求文档（PRD）

**工具：ChatGPT Plan Mode / 其他计划类 Skill**

让 AI 根据压力测试后的想法，直接生成 PRD。用原生的 Plan mode 或各种做计划的 Skill 都可以。

### Step 3：需求对齐

**Skill：Groomy**

> AI 生成的 PRD 肯定有很多地方和你的想法不一致。

Groomy 能帮你问出所有你忽视的细节——前三步走完，你的想法就会非常健壮，不会像"上来直接干"那样干到后面问题越来越多。

### Step 4：确定设计基调

**Skill：Design（如 Frontend Design / UI/UX Pro Max 等）**

一个好的产品需要有好的前端设计。用 Design Skill 为整个产品的各个页面和色调进行设计。

### Step 5：开发 + 全流程质量管理

| **环节**    | **工具/Skill**                  | **作用**        |
| --------- | ----------------------------- | ------------- |
| 🏗 开发     | Codex / Cursor                | 实现功能          |
| 📦 版本管理   | GitHub                        | 每完成一个阶段记得推送   |
| 🐛 Bug 检测 | Hunt                          | 找项目的 Bug      |
| ✅ 测试      | TDD                           | 对项目进行测试       |
| 🔧 重构     | ImproveCoderBase Architecture | 架构需要重构时优化项目结构 |

***

## 工作流全览

```
想法 → Star Pressure Test（压力测试）
     → ChatGPT Plan Mode（生成PRD）
     → Groomy（需求对齐）
     → Design（确定设计基调）
     → 开发（Codex/Cursor）
     → GitHub（版本管理）
     → Hunt（Bug检测）+ TDD（测试）
     → ImproveCoderBase Architecture（架构重构）
```

***

## 思考与延伸

1. **你的 VibeCoding 卡在哪一步？** 很多人跳过前三步直接开干，结果越干问题越多。这套工作流的核心是：先把想法"做扎实"，再动手。
2. **PRD 不是终点，是起点**：Groomy 的价值在于帮你发现"你没想到的问题"——PRD 生成后不进行对齐，才是大多数项目失败的根源。
3. **黑客松验证过的流程**：作者是在两场黑客松 + 一场创意比赛的实际项目中总结出这套流程的，可操作性很强。
4. **可拓展性**：作者说"中间还有很多 Skill"，但这套框架是最具普适性的——可以基于此玩串出自己的工作流。

## 完整文稿

> 为什么你 VibeCoding 出来的就是玩具，大佬做出来的却是成熟的产品？很大的原因在于你工作流程有问题。
>
> 在5月份我倒了两场黑客松、一场创意比赛，我总结出了下面这套极简工作流：
>
> **第一**，我们不要一有想法就马上开始动工。很多想法在刚开始时都是有问题的，所以我们要用 **Star Pressure Test** 这个 Skill 对想法进行压力测试。Claude 也有类似的 Skill，不过我没用过就不具体推荐了。
>
> **第二**，给这个想法写一个产品需求文档。这一步可以让 ChatGPT 根据压力测试的想法直接生成，用原生的 Plan mode 或者各种做计划的 Skill 都可以。
>
> **第三**，AI 生成的初版 PRD 肯定有很多地方和你的想法不一致，这时候我们要和它进行对齐。这里用 **Groomy** 这个 Skill，它能够帮你问出所有你忽视的细节。
>
> 前三步走完，你的想法就会非常健壮了，不会像上来直接干一样，干到后面越来越多问题。
>
> **第四**，要确定设计的基调。一个好的产品需要有好的前端设计，用 **Design** 为整个产品的各个页面和色调进行设计。然后就可以开始动工了。
>
> 动工的过程中，每完成一个阶段记得都要推送到 **GitHub** 进行版本管理。还可以使用 **Hunt** 来找项目的 Bug，用 **TDD** 对项目进行测试。如果架构需要重构，还可以用 **ImproveCoderBase Architecture** 来优化项目解构。这样可以很好的保证产品的可用性。
>
> 其实中间还有很多 Skill，但我觉得这一套工作流是最具有普适性的，可以用到所有的项目上且可拓展性很强。大家可以基于此玩串出一套属于自己的工作流。
