## 一、核心观点与总结

### 核心论点

Superpowers的第一个技能是**Brainstorming（头脑风暴）**，它是一个**Hardgate（硬门槛）**——没有设计文档就不允许写代码，不管项目多简单，这个规则没有例外。核心理念：**设计是给人类看的，代码是给机器执行的**。先跟人类讨论设计达成共识，再让机器执行代码，比直接让AI猜意图写代码再返工，效率高10倍。

### 为什么需要Brainstorming？

**常见问题**：用AI写代码，说"帮我做一个用户登录功能"，AI啪啪啪写出来了，能用。但你不知道它用了什么认证方式、密码怎么存的、错误处理怎么做的。

**核心矛盾**：

* 代码能跑 ≠ 代码对
* 发现设计有问题时，改代码的成本比改设计高出**10倍不止**
* 你觉得你知道要做什么，但**AI不知道**——你不写设计，AI只能猜你的意图

**结论**：没有设计你是在赌AI能猜对你的意图；有了设计你是在确认AI理解了你的意图。一个是赌，一个是确认。

### Brainstorming的9步流程

| **步骤** | **名称**      | **做什么**                              |
| ------ | ----------- | ------------------------------------ |
| 1      | **探索项目上下文** | 检查项目文件结构、文档、最近提交记录，了解现在有什么           |
| 2      | **视觉伴侣**    | 如果涉及UI布局/架构图，询问是否需要浏览器端可视化工具         |
| 3      | **提问**      | 一个一个问题确认需求（不是一次问10个），了解目的、约束、成功标准    |
| 4      | **提出方案**    | 给出2-3个不同方案，每个有利弊分析，AI给出自己的推荐和理由      |
| 5      | **分段展示设计**  | 设计分成几个部分，一部分一部分展示，每部分都问"这样行不行"       |
| 6      | **写设计文档**   | 所有部分确认后，写成正式文档，保存到Docs目录             |
| 7      | **自审**      | AI自己检查：站位符(TBD/TODO)、前后矛盾、范围过大、模棱两可  |
| 8      | **用户审核**    | 交给用户审阅，有修改意见就改，改完再审，直到满意             |
| 9      | **交接**      | 文档通过后，自动调用下一个技能Writing Plans，不是跳到写代码 |

**关键**：需求太大时（如"做一个有聊天、文件存储、账单和分析的平台"），AI会立即告诉你需要先拆成子项目，不会在一个设计里塞十个系统。

### 视觉伴侣的设计原则

**核心原则**：视觉内容用浏览器，文字内容用终端。

| **问题类型**   | **回答方式**      | **示例**       |
| ---------- | ------------- | ------------ |
| **视觉问题**   | 浏览器展示         | "这两个布局哪个更好？" |

AI会提前问你一次要不要用视觉伴侣功能，说不要就用纯文字继续。

### 设计文档规范

**存放位置**：`Docs/Superpowers/Specs/` 目录

**文件名格式**：`日期+主题+design.md`

**内容覆盖**（根据复杂度调整，简单项目几句话，复杂项目好几页）：

* 架构设计

* 错误处理策略
* 测试策略

### 设计原则：隔离与清晰

把系统拆成一个个小单元，每个单元只有一个明确目的，通过清晰接口互相通信，每个单元都能独立理解和独立测试。

**判断标准**（不读代码就能回答的三个问题）：

1. 它能做什么？
2. 怎么用？
3. 它依赖什么？

### 在已有代码库中工作的原则

* **先观察**现有的结构和模式，跟着已有模式走，不要搞特殊
* 如果现有代码有问题影响了你的工作，在设计里加**针对性改进**
* **不要趁机搞大规模重构**，那不是你的目标

### Brainstorming六条核心原则

| **#** | **原则**            | **说明**                         |
| ----- | ----------------- | ------------------------------ |
| 1     | **一次只问一个问题**      | 不要一次问10个把用户搞晕，问清楚再问下一个         |
| 2     | **尽量用选择题**        | 选择题比开放题容易回答，能用选择就不用填空          |
| 3     | **YAGNI（无情的砍需求）** | 不是当前必须的功能全部拿掉，不做预备性设计，不为未来预留接口 |
| 4     | **永远提出两个以上方案**    | 至少两个最好三个，让用户有选择                |
| 5     | **逐步验证**          | 设计一部分确认一部分，不要一口气全写出来再审         |
| 6     | **保持灵活**          | 用户反馈不对马上回去改，设计阶段改主意成本最低        |

**这六条解决了一个核心问题**：让AI从"我来帮你写代码"变成"**我先帮你把事情想清楚**"。

为什么说它容易被忽视？因为大多数人的习惯是有了想法就直接写代码。这个习惯太普遍了，普遍到很多人觉得这才是正常的。但Superpowers说不行，没有设计文档就不允许写代码。

你有没有过这种经历：用AI工具写代码，你说帮我做一个用户登录功能，AI啪啪啪就写出来了，你一看能用。但你仔细想想，你知道它用了什么认证方式吗？你知道密码怎么存的吗？你知道错误处理是怎么做的吗？大概率不知道。这就是直接写代码的问题——代码能跑不等于代码对。更危险的是，当你后来发现设计有问题的时候，改代码的成本比改设计高出10倍不止。

Superpowers的Brainstorming技能要解决的就是这个问题。它的核心理念可以用一句话概括：**设计是给人类看的，代码是给机器执行的**。你跟人类讨论设计，不用一行代码就能达成共识；你跟机器讨论代码，改来改去谁都不满意。

Brainstorming做的事情就是在你写任何代码之前，帮你把需求想清楚、把方案定下来、把设计写出来，你确认了它才往下走。

说到这里要重点说一个概念——**Hardgate（硬门槛）**。在Superpowers里不是所有规则都是建议，有些是不可跳过的强制关卡。Brainstorming就是一个Hardgate。意思是在用户审批设计文档之前，AI不允许做任何实现动作：不能写代码、不能创建项目、不能执行任何命令。注意这个规则没有例外，不管你的项目有多简单，哪怕就是一个to do list、一个配置文件修改、一个工具函数，都必须先出设计文档，拿到用户批准然后才能写代码。

你可能觉得这也太严了，一个五分钟就能写完的功能，为啥要先花10分钟做设计？答案是：因为那五分钟写完的代码，很可能带来50分钟的返工。设计阶段多花10分钟，实施阶段省50分钟，这笔账很清楚。
